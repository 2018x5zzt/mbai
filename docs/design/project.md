# mbai 产品需求文档 & 设计文档 v1.1

- 文档状态：开发就绪
- 目标读者：前端 / 后端开发 Agent
- 核心交付：20 题测试链路 + 12 主人格结果卡 + 8 副标签系统 + 3 彩蛋人格
- 核心判断：**mbai 不是“AI 使用习惯测试”，而是“你和 AI 聊起来到底像什么货色”的赛博嘴脸图鉴**

---

## 1. 项目定义

### 1.1 一句话定义

mbai 是一个测试“你和 AI 聊起来到底像什么货色”的毒舌梗站。

### 1.2 核心洞察

用户测的不是“我是什么性格”，而是“我在 AI 面前到底长什么嘴脸”。

结果必须让用户产生这三种感受：

1. **这也太会看人下菜了吧**
2. **靠，被说中了**
3. **这个我想截图发出去**

### 1.3 产品定位

mbai 当前版本是一个：

> **轻娱乐人格图腾产品**

第一阶段目标不是商业化，而是验证三件事：

- 有没有人愿意测
- 有没有人愿意发
- 有没有结果被说“太像了”

原则是：

> **先证明值得被转发，再证明值得被付费。**

## 2. 当前版本范围

### 2.1 本版本必须包含

- 首页说明页
- 20 题测试链路
- 12 主人格
- 8 副标签
- 3 个条件触发彩蛋人格
- 完整结果页
- 分享卡截图导出
- 人格提示词复制

### 2.2 本版本明确不做

- 付费功能
- 用户系统 / 历史记录
- 社区 / 评论区
- 积分系统
- 后台配置台
- 多语言
- 重动画与复杂互动特效

### 2.3 交互链路

```text
首页
→ 开始测试
→ 20 题单选答题
→ 本地计算人格结果
→ 生成主人格 + 2 个副标签 + 可选彩蛋
→ 展示结果页
→ 截图保存 / 复制文案 / 再测一次
```

目标时长：

- 答题：约 2 分钟
- 结果计算：小于 1 秒

## 3. 产品结构

### 3.1 三层人格结构

| 层级 | 数量 | 作用 | 呈现位置 |
| --- | --- | --- | --- |
| 主人格 | 12 个 | 核心身份图腾 | 结果页主视觉 |
| 副标签 | 8 个 | 恶习叠加 | 结果页副标题 |
| 彩蛋人格 | 3 个 | 条件触发隐藏款 | 结果页特殊角标 |

### 3.2 结果组成

每个用户的结果由以下部分构成：

- `1` 个主人格
- `0~2` 个副标签
- `0~1` 个彩蛋人格
- `1` 条一句话判词
- `1` 组罪状 / 现场还原 / 赦免书
- `1` 条隐藏夸奖
- `1` 组宿敌 / CP 关系
- `3` 条提示词版本

### 3.3 文档拆分

为了便于开发，当前版本的文档按职责拆成 4 份：

- `design/project.md`
  - 产品边界、模块范围、实现策略、验收标准
- `design/mbai-人格分类草案.md`
  - 12 主人格、8 副标签、3 彩蛋人格的完整数据规格
- `design/question-system.md`
  - 20 道题、维度映射、计分算法、接口建议
- `design/result-page-template.md`
  - 结果页结构、分享卡字段、视觉规范

## 4. 品牌与文案原则

### 4.1 品牌关键词

- 毒舌
- 犀利
- 互联网
- 有梗
- 轻快
- 可截图传播
- 伪正经档案感

### 4.2 语气标准

mbai 的语气应该像：

> **一个很懂互联网、很会损你、但损完还给你留台阶的损友。**

标准是：

> **损，但不脏；狠，但能发。**

### 4.3 文案红线

可以做的：

- 戳穿行为模式
- 夸张真实使用场景
- 用“罪状”“现场还原”包装毒舌感

不要做的：

- 过度病理化
- 过度下流
- 让用户产生被羞辱感
- 像医生下诊断
- 引导危险依赖

### 4.4 结果文案公式

结果页文案统一采用：

1. **一句话判词**：先打中
2. **罪状**：再补刀
3. **现场还原**：给画面
4. **赦免书**：给台阶

## 5. 设计方向

### 5.1 视觉方向

当前版本统一采用：

> **伪正经档案风 + 互联网梗图感**

### 5.2 视觉关键词

- 纯白 / 米白底
- 极淡网格纹理
- 黑字为主
- 红色批注为辅
- 粗体无衬线标题
- 像档案批注，不像企业报告

### 5.3 设计目标

结果页和分享卡必须同时满足：

- 一眼看懂人格结果
- 适合截图传播
- 文案可读性强
- 不依赖长文才能成立

设计细节见 `design/result-page-template.md`。

### 5.4 当前前端落地状态

当前代码版本已经把三张核心页面落成统一视觉系统：

- 首页：`杂志封面式 Hero + 样张预览区`
- 答题页：`审问台式双栏结构`
- 结果页：`人格 Hero + 档案主栏 + 传播侧栏`

当前实现重点：

- 首页不再使用均匀信息卡堆叠，而是用大标题直接建立品牌攻击性
- 答题页左侧负责制造节奏感，右侧只负责问题与选项，避免问卷后台感
- 结果页区分 `页面内传播预览` 与 `导出分享海报` 两套展示，不混用
- 移动端保持与桌面同一视觉语言，但压缩模块长度，优先保证首屏人格识别与操作按钮可达

当前前端落地文件：

- `src/main.ts`
- `src/style.css`

## 6. 技术实现策略

### 6.1 数据来源

第一版所有配置均采用静态数据文件：

- 人格数据：静态 JSON
- 副标签数据：静态 JSON
- 彩蛋规则：静态 JSON
- 题库数据：静态 JSON

### 6.2 计算位置

第一版默认采用：

- **前端本地计算**

理由：

- 无需登录
- 延迟更低
- 无需保存用户答案
- 更符合轻娱乐和隐私友好定位

### 6.3 接口策略

如果需要统一封装，也可以保留一个可选接口：

- `POST /api/evaluate`

但该接口本质上只是把前端本地计算逻辑服务化，不应成为 MVP 依赖前提。

### 6.4 推荐实现模块

- `QuestionCard`
  - 单题展示与选项切换
- `ProgressBar`
  - 20 题进度条
- `QuizFlow`
  - 测试流程状态管理
- `evaluateMbaiResult()`
  - 计分与人格计算函数
- `ResultCard`
  - 结果页主内容
- `CoordinateMap`
  - 二维人格坐标图
- `ShareCapture`
  - 截图区域与导出逻辑

## 8. 验收标准

### 8.1 传播性

结果页截图发到微信群 / 小红书时：

- 结果封号一眼可见
- 一句话判词清晰可读
- 副标签和视觉元素不会糊成一团

### 8.2 命中感

找 20 个目标用户测试时：

- 至少 15 人表示“被说中了”

### 8.3 性能

- 从首页到结果页全程不超过 2 分钟
- 结果计算小于 1 秒
- 截图导出在常见手机尺寸上可接受

### 8.4 趣味性

- 副标签叠加描述必须有梗
- 不同人格之间必须有明显区分度
- 文案不能像流水线 MBTI 套壳

## 9. 当前版本一句话总结

> **mbai 第一版不是在做严肃人格测试，而是在做一个能把“你和 AI 的相处嘴脸”一针见血说出来、并且值得截图传播的结果引擎。**
